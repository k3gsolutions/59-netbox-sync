# FASE 3.10 — Web UI Pending Item Editor Modal

**Status:** Complete
**Date:** 2026-04-29

## Goal

Replace the manual CSV handoff with a local modal-based workflow:

- user opens a pending item in the Web UI
- modal renders only the missing fields for that team/object type
- form validates inline
- backend validates again
- backend saves a unified CSV for Week 1 validation
- backend writes a local audit trail
- after save, the local validation pipeline can run safely
- no NetBox writes, no approval creation, no apply/sync

## Routes

### `GET /service-engagement/{device}/pending-items`

Returns HTML or JSON with the current pending queue for the device.

### `GET /service-engagement/{device}/pending-items/{safe_item_id}`

Returns one item plus the dynamic schema used by the modal.

### `POST /service-engagement/{device}/pending-items/{safe_item_id}/response`

Saves the local response only.

Response shape:

```json
{
  "success": true,
  "message": "Resposta salva localmente. Nenhuma alteração foi feita no NetBox.",
  "team": "Service Team",
  "csv_path": "reports/pilot-device-compliance/week1-responses/service-team-response.csv",
  "validation_status": "answered"
}
```

## UX

- pending queue shown on device, responses, week2 candidates, and pending-items pages

  - `Editar pendência`

## Validation Rules

### Service Team

Fields:

- `tenant`
- `service_type`
- `criticality`
- `owner`
- `evidence`
- `notes`
- `status`

If `status=answered`:

- `tenant` required
- `service_type` required and valid
- `criticality` required and valid
- `owner` required
- `evidence` required

If `status=blocked`, `status=rejected`, or `status=needs_clarification`:

- `notes` required

### Network Ops

Fields:

- `ip_address`
- `detected_interface`
- `detected_vrf`
- `interface`
- `vrf`
- `relation_type`
- `service_relation`
- `owner`
- `evidence`
- `notes`
- `status`

If `status=answered`:

- `interface` required only when the collector did not detect one
- `vrf` required only when the collector did not detect one
- `relation_type` required
- `service_relation` required only when `relation_type=service`
- `owner` required
- `evidence` required
- `relation_type=unknown` requires `notes`
- `status=blocked|rejected|needs_clarification` requires `notes`

`ip_address` display rule:

- when the collector detects interface/VRF, the modal pre-fills the fields
- the user may correct the pre-filled values
- if nothing is detected, the UI asks for the missing mapping

### BGP Team

Fields:

- `remote_asn`
- `remote_bgp_group`
- `policy_intent`
- `owner`
- `criticality`
- `evidence`
- `notes`
- `status`

If `status=answered`:

- `remote_asn` required and numeric
- `remote_bgp_group` required and slug-safe
- `policy_intent` required and at least 5 chars
- `owner` required
- `criticality` required
- `evidence` required

### Common Rules

- `updated_by` required
- `updated_at` generated by backend
- secret terms blocked:
  - `token`
  - `password`
  - `secret`
  - `NETBOX_WRITE_TOKEN`
  - `private key`
  - `bearer`
  - `Authorization`
- path traversal blocked in submitted `object_key`
- CSV row updates in place by `object_key`
- audit log is append-only
- `object_key` values with `/`, `\\`, or `..` are rejected

## Storage

- CSV: `reports/pilot-device-compliance/week1-responses/<team>-response.csv`
- Audit: `reports/pilot-device-compliance/week1-responses/audit/<team>-response-audit.json`

The CSV header is unified and compatible with `validate_week1_responses.py`.

## Validation After Save

The Web UI does not run the Week 1 validator automatically.
The newer flow can trigger local validation and the Week 2 gate after save or finalize.

Use:

```bash
python3 tools/local/validate_week1_responses.py \
  --template reports/pilot-device-compliance/week1-metadata-collection-template.csv \
  --responses-dir reports/pilot-device-compliance/week1-responses \
  --output reports/pilot-device-compliance/week1-response-validation.md \
  --device 4WNET-MNS-KTG-RX
```
